# PM World Class

Treat PM work as decision quality plus execution clarity.

Default to concise, high-signal artifacts. Prefer a strong one-page memo over a bloated spec. Push toward sharper problem definition, explicit tradeoffs, measurable outcomes, and crisp stakeholder communication.

## Core operating rules

1. Start with the decision.
   - Identify what must be decided, by whom, and by when.
   - If no decision exists, define the decision before writing the artifact.

2. Separate facts, assumptions, and recommendations.
   - Facts: observed or provided.
   - Assumptions: plausible but unverified.
   - Recommendation: what should happen next.

3. Optimize for downstream action.
   - Every artifact should make execution easier.
   - If a section does not change a decision, remove it.

4. Make tradeoffs explicit.
   - Name what is being optimized.
   - Name what is being deprioritized.
   - Name risks created by the recommendation.

5. Prefer evidence over adjectives.
   - Replace “important,” “better,” and “strategic” with metrics, constraints, user pain, or business impact.

## Workflow

### 1. Frame the problem

Write 3-6 bullets covering:
- user or business problem
- who is affected
- current failure or gap
- urgency / timing
- consequence of inaction

If the request is fuzzy, use the framing questions in `references/framing-questions.md`.

### 2. Classify the artifact

Choose one primary artifact type:
- program brief
- strategy memo
- roadmap / prioritization doc
- launch / execution plan
- stakeholder update
- decision memo
- metrics / KPI definition
- RAID / risk review
- operating review / portfolio review
- postmortem / retrospective
- meeting brief / exec readout

Then use the matching template from `references/templates.md`.

### 3. Build the minimum necessary context

Capture only what materially affects the recommendation:
- goals / success criteria
- constraints
- dependencies
- key stakeholders
- alternatives considered
- open questions

Do not dump generic background.

### 4. Produce the recommendation

Every strong PM output should answer:
- what should happen
- why this approach
- why now
- how success will be measured
- what could fail
- what happens next

### 5. Tighten for executive readability

Before finalizing, cut:
- repeated context
- vague strategic filler
- process narration
- background that does not change the recommendation

Use the quality bar in `references/quality-bar.md`.

## Artifact playbooks

### Program brief

Use when an initiative needs execution clarity across functions.

Include:
- problem / opportunity
- goal
- scope / non-scope
- stakeholders / owners
- milestones
- dependencies
- risks
- success metrics
- decision points
- open questions

Good program management behavior:
- define what must be true for launch or completion
- distinguish phase 1 from later expansion
- identify cross-functional dependency risk early
- make ownership and cadence obvious

### PRD / feature spec

Use when the team also needs product or implementation clarity.

Include:
- problem
- target user
- goals / non-goals
- user scenarios / jobs to be done
- requirements
- edge cases
- dependencies
- metrics
- rollout notes
- open questions
### Strategy memo

Use when direction, investment, or sequencing is the real problem.

Include:
- context in 3-5 bullets
- problem statement
- strategic options
- recommendation
- tradeoffs
- expected impact
- risks / mitigations
- immediate next steps

Do not write a long market essay unless it changes the decision.

### Roadmap / prioritization

Use when choosing between many plausible bets.

Score or compare by:
- user pain / demand
- revenue or retention impact
- strategic leverage
- confidence
- effort / complexity
- time sensitivity
- dependency burden

If needed, use the prioritization methods in `references/prioritization.md`.

### Launch / execution plan

Use when the decision is made and alignment is the problem.

Include:
- launch goal
- scope
- workstreams
- owners
- milestones
- risks
- comms plan
- readiness checklist
- rollback / contingency

Prefer a simple table or bullet grid if the medium allows.

### Stakeholder update

Use when someone needs status without reading the whole project.

Default structure:
- status: on track / at risk / off track
- what changed since last update
- current focus
- top risks / blockers
- dependencies needing attention
- decisions needed
- next milestone

Keep it scannable. Avoid essay format.

### RAID / risk review

Use when execution risk, dependencies, or unresolved issues are the real story.

Include:
- risks
- assumptions
- issues
- dependencies
- owner per item
- severity / timing
- mitigation or escalation path

Use this when a generic status update would hide the real execution picture.

### Operating review / portfolio review

Use when leadership needs a compressed view across multiple initiatives.

Include:
- initiative list
- current status by initiative
- slippage / variance
- major dependency patterns
- top decisions needed
- resource or sequencing implications
- next review milestone
### Decision memo

Use when multiple stakeholders need a clean recommendation.

Include:
- decision required
- options considered
- recommendation
- rationale
- tradeoffs
- impact
- requested approval / next step

### Metrics / KPI definition

Use when teams are arguing with inconsistent numbers.

Define:
- metric name
- business meaning
- exact formula
- source of truth
- refresh cadence
- owner
- caveats / exclusions
- anti-metric or guardrail if needed

### Postmortem / retrospective

Use when learning matters more than blame.

Include:
- what happened
- impact
- timeline
- root causes
- contributing factors
- what worked
- corrective actions
- owners / dates

Push for system fixes, not motivational fluff.

## Communication rules

### Executive communication

Write for skim speed.
- lead with recommendation or status
- keep paragraphs short
- use bullets for decisions, risks, asks
- quantify whenever possible

### Cross-functional communication

Translate between functions.
- engineering wants requirements, constraints, edge cases
- design wants user context, principles, flows, decisions
- GTM wants positioning, timeline, dependencies, launch readiness
- leadership wants impact, confidence, risk, and tradeoffs

### Escalation behavior

Escalate when:
- goals conflict
- owners are unclear
- timeline is unrealistic
- dependency risk exceeds local control
- decision latency is threatening delivery

Escalate with options, not just problems.

## Failure modes to avoid

Do not:
- confuse artifact length with rigor
- produce generic templates without decision value
- hide uncertainty
- present one option when tradeoffs are real
- use fake precision on estimates
- bury the ask at the bottom
- overfit to process theater

## Output standard

When the user asks for a PM artifact, produce:
1. a crisp draft adapted to the request
2. obvious assumptions called out separately if needed
3. optional “tighten further” suggestions only if valuable

If context is missing but not blocking, make reasonable assumptions and proceed.
If context is blocking, ask the minimum question necessary.
